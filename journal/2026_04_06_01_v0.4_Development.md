- All JSON test fixture files: `"archived": false/true` → `"in_trash": false/true`

**4. Version bump**

### Session 2 (April 6, 2026) — Phase 2 Planning + Missing `position` Tests

#### Phase 2 Planning: New Block Types

Researched the three block types for Phase 2. Retrieved the live block reference docs from `developers.notion.com/reference/block`. Key findings:

**`heading_4`**
- Identical shape to `heading_1/2/3`: `rich_text`, `color`, `is_toggleable`, optional `children` when toggleable
- Needs: `Heading4Content`, `Block.Heading4`, `Heading4RequestContent`, `BlockRequest.Heading4`, three `heading4()` overloads in `PageContentBuilder`, validation

**`tab`**
- `tab` property is an empty object `{}`; tab metadata (label, icon, content) lives in paragraph children
- Tab paragraph children support an optional `icon` field (emoji, custom emoji, external, file)
- Plan: `TabContent` empty class, `TabRequestContent(children)`, `Block.Tab`, `BlockRequest.Tab`, `tab()` in DSL
- Add `icon: PageIcon? = null` to `ParagraphContent` (for reading) and `icon: CalloutIcon? = null` to `ParagraphRequestContent` (for creating)
- Add `icon` param to all `paragraph()` overloads in `PageContentBuilder`
- `PageIcon` (from `models/pages/PageIcon.kt`) is the right type for the response model; `CalloutIcon` for requests

**`meeting_notes`** (read-only)
- Renamed from `transcription` in `2026-03-11`
- Upgrade guide shows content is `{ "rich_text": [...] }` — simple paragraph-like structure
- `ignoreUnknownKeys = true` in `NotionClient.kt` handles any additional live API fields gracefully
- NOT added to `BlockRequest` or `PageContentBuilder` — API does not allow creation

#### Approach: One Block Type at a Time

Phase 2 will be implemented in three focused sub-tasks, each covering all layers (model, request, DSL, tests) for one block type before moving to the next:

1. `heading_4` — minimal blast radius, good warm-up
2. `tab` — requires ParagraphContent/ParagraphRequestContent changes too
3. `meeting_notes` — response model only

#### Gap Identified: Missing Tests for `BlockAppendPosition`

Phase 1 implemented the `position` parameter for `appendChildren` (`BlockAppendPosition` sealed class with `AfterBlock`, `Start`, `End`), but no unit or integration tests were written to verify this behavior. Before or alongside Phase 2, we should add:

- **Unit test** (in `unit/api/BlocksApiTest.kt` or a new `BlocksApiPositionTest`): mock response, verify that the `position` object is serialized correctly in the request body for all three variants (`AfterBlock`, `Start`, `End`)
- **Integration test** (in `integration/`): create blocks in a test page, then use `appendChildren` with `position = Start` and `position = AfterBlock(...)` to verify ordering is respected by the live API

This should be done before Phase 2 to close the gap in coverage from Session 1.

### Session 4 (April 6, 2026) — Rename `archive()` → `trash()` + Doc Updates (Phase 1 Docs)

#### Background

While beginning the doc update pass for Phase 1, it became clear that the `archive()` method name retained in Session 1 was a poor choice. The official Notion API (`2026-03-11`) removed `archived` entirely in favour of `in_trash`, and the official docs use "trash" as the primary verb throughout. Keeping `archive()` in our library API would have confused users familiar with the official docs.

#### Decision Reversed

The Session 1 key decision — *"Public `archive()` method name kept"* — was reversed. The correct naming throughout is `trash()`.

#### Changes Made

**Renamed in source:**
- `PagesApi`: `suspend fun archive(pageId)` → `suspend fun trash(pageId)`; KDoc updated
- `DatabasesApi`: `suspend fun archive(databaseId)` → `suspend fun trash(databaseId)`; KDoc updated
- `UpdatePageRequestBuilder`: `fun archive(archived: Boolean = true)` → `fun trash(inTrash: Boolean = true)`; KDoc and class-level doc updated
- `DataSourceRequestBuilder`: `fun archive()` → `fun trash()`; `fun unarchive()` → `fun restore()`

**Renamed in tests and examples (all call sites):**
- All `notion.pages.archive(...)` → `notion.pages.trash(...)`
- All `notion.databases.archive(...)` → `notion.databases.trash(...)`
- All `client.pages.archive(...)` → `client.pages.trash(...)`
- All `client.databases.archive(...)` → `client.databases.trash(...)`
- All `archive()` DSL calls in `UpdatePageRequestBuilderTest`, `PagesExamples`, `UpdatePageRequestBuilderIntegrationTest` → `trash()` / `trash(false)`
- Test names and log messages updated throughout (~25 files total)
